# TCATemplate

A minimal iOS template app built with [The Composable Architecture (TCA)](https://github.com/pointfreeco/swift-composable-architecture), with **architecture rules enforced by static analysis from day one**.

The project layout follows the `try-swift-tokyo` style split:

- `App/` — the iOS application target (generated by XcodeGen); a thin composition root
- `iOS/` — a Swift package containing all TCA features, clients, and tests
- `TCAArchRules/` — a reusable Swift package of architecture lint rules (built on [Harmonize](https://github.com/perrystreetsoftware/Harmonize))

## What's included

- **Counter tab** (`CounterFeature`) — a complete example of the enforced patterns: structured actions, an effect calling a `@DependencyClient` (`NumberFactClient`), and a `TestStore` test with dependency overrides
- **Navigate tab** (`SecondTabFeature`) — a B → C → D drill-down navigation example using tree-based navigation
- **Reducer tests for every feature** (Swift Testing + `TestStore`)
- **Architecture tests** (`ArchitectureTests`) — 20 structural rules that fail the test suite on violation
- **SwiftLint config** — curated opt-in rules plus custom hygiene rules (no `print`, no raw `Task {}` in features, no nondeterministic `Date()`/`UUID()`, and more)

## Setup

```bash
brew install xcodegen swiftlint
cd App
xcodegen generate
open TCATemplate.xcodeproj
```

> ⚠️ Xcode 27 beta cannot build some dependencies. From the command line, prefix commands with
> `DEVELOPER_DIR=/Applications/Xcode-26.5.0.app` (or any Xcode 26.x install).

## Running the checks

| Layer | Tool | Command |
|---|---|---|
| Architecture (structure, placement, coverage) | Harmonize via `ArchitectureTests` | `swift test --package-path iOS --filter ArchitectureTests` |
| Syntax & hygiene | SwiftLint | `swiftlint lint --quiet --lenient` |
| Everything (unit + architecture tests) | Swift Testing | `swift test --package-path iOS` |

## Architecture rules

Apps created from this template start with **empty baselines (zero known violations). Keep them empty** — fix the code instead of silencing rules.

The main rules (see `docs/architecture-rules.md` for the full list and rationale, in Japanese):

- `@Reducer` types are named `*Feature`; one reducer per file; reducers and views live in separate files
- Actions are structured into `view` / `internal` / `delegate` / `binding` / child-reducer cases
  (`@CasePathable` nested enums, `ViewAction` conformance, `@ViewAction(for:)` on views)
- `State` is `@ObservableState` + `Equatable`
- Presentation goes through a `@Reducer enum Destination` (bare `@Presents` vars are not allowed)
- Clients are named `*Client`, declare explicit `liveValue` / `testValue` / `previewValue`, and never import `ComposableArchitecture`
- No singletons (`.shared`) and no direct `*Manager` access — everything is injected via `@Dependency`
- Feature code does not access persistence, OS, or external SDK APIs directly — use `@Shared` or a Client
- `Store(initialState:)` is created only in the composition root (`AppRootView`) and `#Preview`
- Every reducer has a matching `<ReducerName>Tests` suite

Violations fail the test suite with an issue attached to the offending source line. If a rule ever needs a permanent exception (e.g. an Apple API that requires `.shared`), register it in `ArchitectureTests.swift` with a comment explaining why — never by adding baseline entries.

## Adapting this template for a new app

1. Rename the app target and bundle identifier in `App/project.yml`, then `xcodegen generate`
2. Add features as new SPM targets under `iOS/Sources/<Name>Feature` (reducer + view in separate files)
3. Add clients as `<Name>Client` targets with the three dependency values
4. Run `swift test --package-path iOS` — the architecture rules will guide the rest
